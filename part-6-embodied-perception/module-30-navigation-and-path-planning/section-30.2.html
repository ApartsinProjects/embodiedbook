Section 30.2: Graph search: BFS, Dijkstra, A*

Graph search is a contract between the map discretization and the motion the robot can actually execute.

A Local Planner With Commitment Issues

This section assumes familiarity with the navigation problem framing introduced in section 30.1, particularly the distinction between map representation and motion execution. The graph-search guarantees developed here, especially admissibility and consistency of A*, are extended in section 30.3, where sampling-based planners (RRT, RRT*, PRM) handle continuous configuration spaces that cannot be discretized into grids. The cost-function and heuristic design ideas recur in Part VII alongside learned reward shaping for reinforcement-based navigation.

Big Picture

A warehouse robot receives a "go to shelf 47" command. Its map is a grid of 10,000 cells. BFS would blindly fan out in all directions, Dijkstra would respect aisle penalties, but A* reaches the shelf having examined fewer than a tenth of the cells, because a single admissible number (straight-line distance remaining) prunes every clearly-wrong direction before it is ever explored. That pruning is why every deployed mobile robot from surgical arms to delivery drones runs some variant of A* at its core. Here you will build all three algorithms from scratch, prove the admissibility condition that makes A* optimal, and measure exactly where each algorithm saves or wastes work on real grid maps.

Problem First

A robot has a map. It has a current position and a goal. The map is not a path: it is a representation of what space exists, not a prescription for how to move through it. Graph search is the step that converts a static map into an ordered sequence of waypoints the robot can actually follow. Without this step, the map is inert data.

The three algorithms in this section represent a progression of assumptions about the world. Breadth-first search (BFS) is appropriate when all moves cost the same, for instance on a uniform grid where every step is identical. Dijkstra's algorithm relaxes that assumption and finds the cheapest path when edges carry different costs, such as when moving through mud costs more than moving on pavement. A* adds one more ingredient: what engineers call the admissible distance shortcut, a heuristic estimate of remaining distance to the goal that lets the search skip large portions of the map that are clearly in the wrong direction. Each step in this progression trades a simpler assumption for better real-world applicability.

A* is correct when the heuristic is admissible, meaning it never overestimates the true remaining cost. Consistency gives an even stronger property: once a node is expanded, its best cost is known. Those guarantees matter when the graph stands in for physical space.

Consider a specific case: a 5-by-5 grid where a warehouse robot must travel from cell (0,0) to cell (4,4). BFS expands cells in rings outward, visiting up to 25 cells before finding the goal. Dijkstra with uniform cost does the same work but is ready to handle a variant where the center row carries a penalty cost of 3 because it crosses a slope. A* with Euclidean distance as its heuristic skips most cells on the left and bottom edges entirely, expanding roughly 9 cells before reaching (4,4), because the heuristic steers it diagonally toward the goal from the start. The path length is identical in all three cases on this open grid; the difference is purely in how many cells were examined and therefore how long planning took.

Feasibility Before Beauty

A shortest path is only meaningful on a graph that matches the robot body.

Formal Model

Most navigation methods can be read as constrained search or optimization:

$$ f(n)=g(n)+h(n),\quad 0\le h(n)\le h^*(n) $$

The cost term is graph distance or risk-weighted distance; constraints are traversable cells, obstacle inflation, motion connectivity, and heuristic admissibility.

Algorithm: Section 30.2 Planning Loop
  1. Build nodes from grid cells, topological places, or lattice states.
  2. Assign edge costs for distance, clearance, slope, turn penalty, and risk.
  3. Choose a heuristic whose units match the edge cost.
  4. Audit the found path against collision checks and controller feasibility.

Worked Diagnostic

Code Fragment 1 isolates graph-search invariants: frontier update, predecessor map, path cost, and heuristic behavior. The point is to verify the path object before handing it to a controller.

# Compute A* priorities for three frontier nodes.
# The admissible heuristic focuses search without changing cost units.
frontier = [
    {"node": "A", "g": 4.0, "h": 6.0},
    {"node": "B", "g": 7.0, "h": 2.0},
    {"node": "C", "g": 5.0, "h": 5.0},
]
ranked = sorted((n["g"] + n["h"], n["node"]) for n in frontier)
print(ranked)
print(f"expand={ranked[0][1]}")
[(9.0, 'B'), (10.0, 'A'), (10.0, 'C')] expand=B

Expected output interpretation. The frontier printout shows that node B is expanded first because its combined score $f=g+h$ is smallest, even though it is not the node with the smallest traveled cost alone. Nodes A and C tie at 10.0, which is a useful reminder that tie-breaking policy can still influence search order and runtime even when optimality is unchanged.

Code Fragment 1: The frontier ranking shows how A* combines traveled cost and heuristic cost. Node B expands first because its total priority is lowest, not because it is nearest in Euclidean distance alone.

When using Python's heapq for A*, add a secondary tie-breaker as the second tuple element, for example (f, node_id), so that equal-priority nodes are always expanded in a deterministic order. Without this, Python will attempt to compare node objects directly and raise a TypeError the moment two f-scores collide. In Nav2, the equivalent parameter is use_final_approach_orientation combined with a consistent cost map resolution; mismatched resolutions between the planner costmap and the controller costmap are the most common source of paths that look correct in RViz but cause the robot to stall at the first waypoint.

Tool Workflow

Library Shortcut

NetworkX can teach graph search in a few lines, while Nav2 global planner plugins apply the same idea to costmaps and robot goals. The shortcut saves manual priority queues, path reconstruction, and middleware integration.

Keep the tiny graph-search implementation as a test for admissibility, cost accumulation, and predecessor reconstruction. Use Nav2 global planners or OMPL when maps and robot constraints get large.

Failure Mode To Test

Replay a blocked corridor, wrong map cell, diagonal-motion mismatch, stale costmap, and localization offset. Graph search fails differently when the graph is wrong versus when execution is wrong.

Practical Example

Log open set size, closed set, predecessor map, final path, costmap snapshot, and controller tracking error. That evidence separates search quality from downstream execution.

Integration Checklist

Freeze grid resolution, obstacle inflation, heuristic, tie breaking, start-goal convention, and cost encoding before comparing BFS, Dijkstra, and A*. Small convention changes can dominate the result.

Research Frontier

The frontier includes dynamic graph search, learned heuristics with admissibility checks, and multi-resolution planners that combine topological maps with local geometric grids.

Memory Hook

Self Check

Can you state the search space, cost function, constraints, replanning trigger, controller interface, and failure metric for graph search: bfs, dijkstra, a*? If not, the planner is not specified enough to deploy.

Key Takeaway

Graph search: BFS, Dijkstra, A* is ready for embodied use when route quality, dynamic feasibility, local control, and recovery behavior are measured in the same replay.
