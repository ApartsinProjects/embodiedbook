Section 45.4: Terrain adaptation, parkour, and rapid motor adaptation

A Careful Control Loop
Big Picture

Terrain adaptation, parkour, and rapid motor adaptation is a concrete locomotion and mobility skill. The page treats it as an embodied loop with named observations, actions, physical constraints, metrics, and recovery behavior.

What This Section Develops

This section develops terrain adaptation and parkour as a concrete embodied AI skill rather than a label. The core contract is: observe height map, foothold affordances, body velocity, and disturbance history, adapt the policy online to terrain and payload changes, and judge the result with obstacle completion and adaptation latency.

The reader should connect this contract to earlier material on coordinate frames, control foundations, simulation stack. Those links matter because Part IX skills fail at interfaces: a good model with the wrong frame, timing budget, or evaluation panel still produces a fragile robot.

Action Is The Test

Locomotion is controlled falling with evidence. The agent must keep balance while the ground, payload, and timing keep changing. Theory

The minimal formal object is a partially observed control loop. At time $t$, the agent receives $o_t$, estimates $\hat s_t$, chooses action $a_t$, and receives a new observation $o_{t+1}$. In Terrain adaptation, parkour, and rapid motor adaptation, the important state variables are height map, foothold affordances, body velocity, and disturbance history, while the action interface is to adapt the policy online to terrain and payload changes.

The method is useful only if it improves a construct-matched measurement: obstacle completion and adaptation latency. That metric must be co-computed for the baseline and the library shortcut on the same scenario panel, with the same seed policy and the same termination rules.

Mechanism

The mechanism is not a black box. It is a chain of observation encoding, state estimation, constraint checking, action selection, execution, and verification. Each link should emit a log record so a failure can be traced to perception, state, planning, control, timing, data coverage, or evaluation.

Common Failure Mode

The common mistake is to evaluate a component in isolation and then assume the closed loop will inherit that score. Embodied systems often fail at the interfaces between components.

Practical Example
