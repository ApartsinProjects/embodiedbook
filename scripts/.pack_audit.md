# Meta-Agent Audit: Publication Pack Run (2026-07-02)

Book-level run across all 7 passes on the EmbodiedAI book (60 chapters, 379 sections).
Scope of this run's diff: 296 files touched (295 content/CSS files + 1 new audit file),
spanning all 12 parts plus `styles/book.css`.

## Pack with most skipped tasks

**Pass 1 (Deduplication, Agent #42).** Of 23 unique duplicate-paragraph pairs flagged by
`detect_duplicates.py` at similarity >= 0.55, only 5 pairs (29.6/29.7/29.8, 30.6/30.7,
34.6/34.8) were genuine localized duplication and got MERGE INTO / DIFFERENTIATE fixes.
The remaining ~19 pairs (the majority) were correctly triaged as SKIP: they are a
book-wide parameterized template ("sits inside the Part II robotics contract...",
"When X fails, avoid labeling the whole method as weak...", "becomes useful when it is
tied to a closed-loop contract...") reused 20-53 times across the book with only the
section topic name substituted. This is intentional scaffolding, not a defect, but the
dedup script's Jaccard-similarity signal cannot distinguish "same template, correctly
parameterized" from "same content, lazily duplicated" without a human/agent read of
actual prose content. Net effect: most of Pass 1's raw hit list was noise relative to
the pack's own MERGE/CONSOLIDATE/DIFFERENTIATE/SKIP taxonomy, and the agent spent most
of its time triaging rather than fixing.

## Pack with most edits per section (potential over-engineering)

**Pass 4 (Bibliography, Agent #35)** had the largest mechanical footprint: 115 of 379
section files were rewritten to re-sort bibliography entries by year descending. This
is a legitimate, scoped fix (three different bibliography HTML templates existed in the
book: `<h2>Section References</h2>`, `<div class="bibliography-title">`, and
`<h2>Bibliography &amp; Further Reading</h2>`, and ~192 sections had entries in the
wrong order), but it is also the pass most exposed to silent corruption risk: a single
regex bug in the first version of the sort script silently matched 0 files for two of
the three template variants and would have under-reported "unsorted" sections as
"fixed" if not re-verified end-to-end. The fix that shipped re-validates structural
integrity (single `<section class="bibliography">`, single closing tag, all files still
end in `</html>`) after the rewrite; that verification step should be a permanent part
of any bibliography-touching pack, not an optional afterthought.

## Suggested prompt improvements for top 2 underperformers

**Pass 1 (Deduplication):** Add an explicit pre-filter step to the prompt: *"Before
triaging any pair, check whether the duplicated string also appears in >= 10 other
sections book-wide (`grep -rl <exact duplicate string> --include=*.html`). If so, treat
it as an intentional book-wide template and mark SKIP without further per-pair review.
Only apply MERGE INTO / CONSOLIDATE / DIFFERENTIATE to pairs where the duplicated text
is unique to that pair (or a small cluster within one chapter)."* This would have let
the agent skip roughly 80% of the raw report immediately instead of manually confirming
each cluster's prevalence one grep at a time.

**Pass 4 (Bibliography):** The prompt's instruction ("Ensure entries are sorted by year
descending") assumes one bibliography HTML template. In practice this book (and likely
future books built by the same pipeline) uses multiple interchangeable templates for
the same semantic content. Add: *"Before writing any bibliography-parsing regex,
enumerate every distinct bibliography section header pattern in use book-wide
(`grep -o` on the `<section class="bibliography">` opening tags and the ~3 tokens that
follow), and confirm the parser matches 100% of sections before applying any bulk
edit. A parser that silently matches 0% of files for a template variant will report
false 'already compliant' results for real violations."* This generalizes beyond
bibliography sorting: any HTML-structure-dependent regex pass on this book should
open with a template-census step, since parts of the book were clearly built by
different agent waves with different scaffolding conventions.

## Secondary findings worth carrying into the next book run

- Pass 2 (Structural Review) found 36 of 60 chapters (60%) missing the `<div
  class="whats-next">` prose block on their last section, despite 100% correct nav-footer
  `next` links. A lint check for `class="whats-next"` presence on every module's terminal
  section, run as a cheap CI-style gate after each content-generation wave, would have
  caught this at the point of authorship rather than at final-pack time.
- Pass 3 (Epigraph) found 36 of 60 chapter epigraphs sharing one verbatim placeholder
  quote/attribution ("An agent becomes interesting at the exact moment the world refuses
  to be a dataset." / "A Patient Embodied AI Agent"), plus a second smaller cluster of 4
  (Part VI) sharing a variant of the same sentence, plus 2 further exact duplicate pairs
  (chapter 1's non-persona "Chapter 1" citation, and chapters 38/39 sharing one quote).
  A per-chapter epigraph uniqueness check (same grep-for-prevalence technique as the Pass 1
  suggestion above) belongs in the epigraph-writing agent's own self-check step, not
  discovered only at final publication pack time.
